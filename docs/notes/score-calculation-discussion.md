# スコア算出方法の議論メモ

日付: 2026-03-12

## 背景

実データを流し込んだ結果、以下の問題が顕在化した:

- **音楽（Last.fm）の存在感が大きすぎる** — ながら聴きで常に積み上がるため、面グラフで他ソースを圧倒する
- **コーディング（トークン数）が爆発する** — トークン数は投入時間と比例しない

## 根本的な気づき: 可処分時間の排他性

データソースの中には「可処分時間を食い合うもの」と「そうでないもの」が混在している。

### 排他的な活動（その間は他のことができない）
- Google Calendar（人と会う）
- Strava ライド（運動中）
- 映画、ライブ

### 並行可能な活動（ながらでできる）
- Last.fm（音楽）— 自転車中も仕事中も聴ける
- Instagram / Twitter（スキマ時間）
- ポッドキャスト

### 食い合いの方向性
- 食い合いはペアごとに異なる（ライドは読書を押し出すが音楽は押し出さない）
- N×Nマトリクスは組み合わせ爆発するため、ソースごとの1次元の係数に落とす方が現実的

## 検討した係数設計

### 案1: 排他性係数を0〜1で設定（並行活動を抑制）
- 音楽: 0.2、ライド: 0.9、カレンダー: 1.0
- **問題**: 既存の自発性係数（spontaneity_coefficient）と掛け合わせると効果が二重になりすぎる
  - 例: lastfm = 0.6(自発性) × 0.2(排他性) = 0.12 ← 潰れすぎ

### 案2: 排他性ボーナス（1.0以上の値で排他的活動を持ち上げる）
- 音楽: 1.0、ライド: 1.2、カレンダー: 1.3
- **利点**: 自発性係数と掛けても下方に二重で潰れない
- **問題**: 音楽の絶対値が大きい問題は解決しない（他を持ち上げても音楽の面積は変わらない）

### 案3: 1.0を基準に上下に振る（0.x 〜 1.x）
- 並行活動は1.0未満（軽く抑える）、排他活動は1.0以上（軽くボーナス）
- **問題**: 結局自発性係数との複合効果の懸念が残る

## 未解決事項: 自発性係数の妥当性

- spontaneity_coefficient はまだ有用かどうか判断がついていない
- 妥当性が不明な係数の上にさらに係数を重ねると、何が効いているか切り分けられなくなるリスク

## 今後の方針: 対話的な係数チューニング

### アプローチ: Human-in-the-loop での探索

事前に正解ラベルをまとめるのではなく、**AIと対話しながら係数を調整する**:

1. AI が係数の組み合わせを変えてスコア・グラフを算出して提示
2. 本人がグラフを見て「この時期は低調だったはずなのに高い」「ここは合ってる」とフィードバック
3. AI がフィードバックをもとに係数を調整して再提示
4. 繰り返して収束させる

**この方法の利点:**
- グラフを見ながら記憶を掘り起こせる（事前にラベルを網羅する必要がない）
- 「なんとなく違う」という曖昧な感覚も拾える
- 自発性係数と排他性係数のどちらが効いているか、オン/オフしながら切り分けられる

### 実装に必要なもの
- 係数を変えた場合のスコアを再計算する仕組み（過去データ全期間）
- 結果をグラフとして提示する手段
- 係数セットの保存・比較機能（「さっきの方がよかった」に対応）

### 別案: 正解ラベル方式による自動探索（Claude提案）

Human-in-the-loop と併用または単独で使える手法:

1. 過去の特定期間に「正解ラベル」を与える（低調/普通/充実 等）
2. ラベルとスコアの相関が最大になる係数の組み合わせをAIが自動探索
3. 数年分のデータ × 数十点のラベルがあれば十分な精度で探索可能

```
正解ラベルの例:
2024-03:  低調（仕事が忙しくて何もできなかった）
2024-08:  充実（夏休みでライブ3本、本も読んだ）
2024-12:  普通
```

**利点:** 対話的チューニングで得たフィードバックをラベルとして蓄積し、後から自動探索に回すこともできる。両方のアプローチは補完関係にある。

## 技術的負債: minutes カラムの問題

当初「すべて分換算」を目指して `activity_records.minutes` を設けたが、実際には分に変換できないソースが多い:

| ソース | minutes の中身 | 実態 |
|--------|---------------|------|
| Last.fm, Strava, SNS | 実際の時間(分) | 自然 |
| Oura readiness/sleep | スコア(0-100) | スコアを分として格納 |
| kashidashi_cd | 枚数 | 3枚 = 3分？ |
| intervals.icu, Gmail, OpenAI/Claude, GitHub, gcal | 0 | 分に変換不能 |

**現状のスコアリングの fallback:**
```python
SUM(CASE WHEN minutes > 0 THEN minutes ELSE raw_value END)
```

基準値で正規化されるため計算結果自体は壊れていないが、コードの意味が曖昧。
**修正案:** スコアリングで `raw_value` を一貫して使い、`minutes` は参考情報に留める。

## 現行スコアリングモデルへの根本的な指摘（Claude提案）

### 指摘1: 達成率と変化検知は別物

現状のスコアは「基準値に対する達成率」だが、このアプリが本当に検知したいのは「自分自身の過去からの変化」。

- 達成率: 700分/週が基準で今週500分 → 71点
- 変化検知: 直近3ヶ月ずっと600分台だったのが今週300分に落ちた → 異常

もともと音楽を1日3時間聴く人が2時間に減っても「まだ基準超え」で見逃す。検知すべきはその**減り方**。

### 指摘2: ベースライン按分が検知したいシグナルを消している

按分の本来の目的は「週途中でスコアがおかしくならないこと」だった。
しかし現在の按分ロジック（データがある日数で基準値を按分）は、
「7日中4日データがない」こと自体が異常シグナルなのにそれを隠してしまう。

**→ 按分ではなく別のアプローチが必要:**
- 期間完了前は暫定表示にする
- 経過日数分のみで比較する
- 等

### 指摘3: 平均がシグナルを潰す

健康指標 = ベースラインソースの平均スコア。1つのソースが爆発すると他のシグナルを覆い隠す:

```
音楽: 180点（爆聴き）、通勤: 0点（在宅）、SNS: 150点（ダラダラ）、Oura: 60点（睡眠悪い）
→ 平均: 97.5点 → NORMAL
```

家にこもって音楽聴きながらSNS、睡眠も悪い — これは「NORMAL」ではない。

### 指摘4: イベント指標が健康判定に反映されない

読書やライブは「イベント」分類のため健康指標に入らない。
しかし「趣味への興味を失う」のはメンタルの重要な兆候。
本を読まなくなった、ライブに行かなくなった — これは健康シグナルとして拾うべき。

## 提案: スコアリングを2層に分離する（Claude提案、合意済み）

### レイヤー1: 可視化（面グラフ・折れ線）

- 各ソースの raw_value をそのまま、あるいは軽い正規化で表示
- 目的は「何がどれだけ起きているか」の事実の提示
- 排他性係数で面グラフのバランスを調整する（見た目の問題）
- 係数チューニング（HITL / 正解ラベル）はこのレイヤーに適用

### レイヤー2: 健康シグナル（ステータス判定）

- 各ソースの「自分自身の過去との乖離」を検知する
- 固定基準値ではなく、直近N日の移動平均からの偏差を使う
- 「下がった」だけでなく「偏った」（音楽だけ突出、他が沈黙）も検知
- イベント指標も「最後にライブに行ってからN日経過」等の形で健康判定に組み込む

### 元のモデルとの共存

2層分離は元のモデル（基準値ベースのスコアリング）を捨てるのではなく、
**元のモデルがレイヤー1になる**形で共存する:

- レイヤー1: `score = (raw / base_value) × 100 × 係数` → 面グラフ・カードの表示用
  - 排他性係数、自発性係数、基準値設定、HITLチューニングはすべてここで有効
  - 100点基準ラインも引き続き意味を持つ
- レイヤー2: 新規追加 → ステータス判定（NORMAL/CAUTION/CRITICAL）はこちらで決める
  - 元のモデルに「できないこと」だけを新レイヤーに任せる切り分け

### この分離の利点

- 排他性係数や自発性係数はレイヤー1（表示）の問題に閉じる
- 健康判定はパターンの変化で判断するので、基準値の恣意性に依存しない
- 音楽の爆発は表示上の調整で済み、健康判定を歪めない

### 実験環境

本番サーバーにデータあり、Claude Codeで実装しているため
GitHubでブランチを切ってUIを使い回しながら実験可能。

- 行動パターンの変化（移動、スマホ使用、睡眠、通話等）から精神状態を推定
- 再発の2週間前に行動異常の検出率が71%高くなるという知見あり

**このプロジェクトとの関係:** 手法は同じだが、データソースが異なる。

## 追加データソースの提案（Claude提案）

### 優先度高: 既存データから抽出可能（新規API不要）

今のソースは「何をどれだけやったか」は捉えているが、
「いつもの時間にいつものことをしているか」（生活リズムの規則性）が弱い。
生活リズムの乱れはメンタルの重要な兆候であり、レイヤー2（変化検知）と特に相性が良い。

| ソース | 何がわかるか | 取得方法 |
|--------|-------------|---------|
| **Oura の起床/就寝時刻** | 睡眠リズムの規則性（スコアより重要かも） | 既に oura_daily にデータあるはず |
| **Last.fm の時間帯分布** | 深夜3時に音楽聴いてる = 生活リズム崩壊 | 既存データの scrobble 時刻から抽出可能 |
| **スクリーンタイムの時間帯** | 深夜のSNS使用 | iOS Shortcut で時間帯別に送れるなら |

### 優先度中: 「外に出ているか」の検知

引きこもりの検知は digital phenotyping の中核。
Strava 通勤で部分的に捉えているが、自転車に乗らない日の外出はわからない。

| ソース | 何がわかるか | 取得方法 |
|--------|-------------|---------|
| **Google Maps タイムライン** | 外出頻度・行動範囲の広さ | Google Takeout or API |
| **Suica/PASMO 利用履歴** | 電車に乗ったか（外出の代理指標） | モバイルSuicaのメール転送等 |
| **Apple Health の歩数** | 最低限の身体活動 | iOS Shortcut webhook |

### 優先度低: あると面白いもの

| ソース | 何がわかるか |
|--------|-------------|
| **Spotify の音楽特徴量** | 聴いてる曲の valence（明暗）の変化 — 暗い曲ばかり聴き始めたら兆候 |
| **YouTube 視聴履歴** | 受動的なコンテンツ消費の量と傾向 |
| **天気データ** | スコアのソースではなく、変化の**文脈情報**（雨の週に活動減は自然 → レイヤー2の偽陽性を減らせる） |

## 排他度マトリクスの実データ検証

ソース間の食い合いの係数を手で決める代わりに、実データから相関行列を出せば答え合わせになる。
ただし、ソースごとにデータがある期間がバラバラで、共通で使える期間が短くなるリスクがある。
**全ソースのデータが十分に揃ってから着手すべき。**

GPUパワー（RTX 3060）が活きるのは、多変量時系列の異常検知（Autoencoder等）を
本格的にやる段階。現状のデータ規模ではCPUで十分。

## 数学的手法の適用案（Claude提案）

### 1. 対数変換（log） — レイヤー1、今すぐ適用可能

人間の感覚は対数的（ウェーバー・フェヒナーの法則）。100→200分の差は大きく感じるが
1000→1100分は感じない。音楽やトークンの爆発を「人間の感覚に合わせる」という理屈で抑制。

```python
# log1p(x) = log(1 + x) でゼロでも壊れない
score = (log1p(raw_total) / log1p(base_value)) * 100 * coeff

# 効果:
# raw=100,  base=700 → 70.5点（線形だと14点）
# raw=700,  base=700 → 100点（基準ぴったり）
# raw=2000, base=700 → 116点（線形だと285点 → 自然に抑制）
```

**実装:** `source_settings` に `log_scale` フラグ追加。lastfm, claude, openai でオン。
`scoring.py:134` と `aggregation.py:196` の計算式で分岐。

### 2. 指数減衰 — イベントソース、今すぐ適用可能

イベントの「鮮度」を表現。昨日のライブと3ヶ月前のライブが同じ重みなのは不自然。
intervals.icu の CTL/ATL と同じ仕組み。

```python
weight = exp(-0.693 * days_ago / half_life)

# half_life=30日の場合:
# 昨日:      weight = 1.00
# 2週間前:   weight = 0.71
# 1ヶ月前:   weight = 0.50
# 3ヶ月前:   weight = 0.12
```

**実装:** `source_settings` に `decay_half_life` 追加。
gcal_live, gcal_holiday, kashidashi 等のイベントソースに設定。
`scoring.py` の `calculate_source_score` でレコードごとに減衰重みを適用。

### 3. エントロピー — レイヤー2（偏り検知）、ソース3-4個で可能

活動の多様性を 0〜1 で数値化。「音楽だけ突出、他が沈黙」を定量的に検知。

```python
# 各活動の割合を出して entropy = -Σ(p * log(p))
# max_entropy で割って 0〜1 に正規化

# 音楽100, 読書30, ライド50, 映画20 → 0.88（バランスよい）
# 音楽200, 読書0,  ライド0,  映画0  → 0.00（一極集中）
```

**実装:** `scoring.py` の `calculate_scores` に `diversity_score` として追加。
フロントに渡してダッシュボードに表示。

### 4. Zスコア — レイヤー2（異常検知）、各ソース3ヶ月分のデータ必要

「自分にとっての異常」を定義。ソースごとのスケールの違いを吸収。

```python
z = (今週の値 - 自分の平均) / 自分のばらつき(標準偏差)

# Z = -2.0 → 下位2.5%のレア度で低い → 異常
# Z =  0.0 → いつも通り
# Z = +2.0 → 異常に高い（躁状態の兆候？）
```

音楽700分(平均600, σ=100) → Z=+1.0 と、読書3冊(平均2, σ=1) → Z=+1.0 が
同じ「自分比で1σ高い」として比較可能になる。

### 5. パーセンタイル — Zスコアの簡易版、同じデータ要件

固定基準値の代わりに「自分の過去データの中で今週は何番目か」。
Zスコアより直感的で説明しやすい。

```
パーセンタイル 5  → 過去の自分の中で下位5% → 異常
パーセンタイル 50 → ちょうど真ん中 → いつも通り
```

### 適用タイミングまとめ

| 手法 | レイヤー | 今すぐ？ | 変更規模 |
|------|---------|---------|---------|
| **対数** | L1（表示） | すぐ可 | source_settings に1カラム + 計算式分岐 |
| **指数減衰** | L1（イベント） | すぐ可 | source_settings に1カラム + 新ロジック |
| **エントロピー** | L2（偏り） | ソース3-4個で可 | 関数1つ追加 |
| **Zスコア** | L2（異常検知） | 各ソース3ヶ月後 | 新関数 + 過去スコア算出 |
| **パーセンタイル** | L2（異常検知） | 同上 | 同上 |

**特に対数と指数減衰は、排他性係数の議論を待たずに音楽の爆発問題を緩和できる。**

## データ量の現実的な制約

上記の手法のうちZスコア・パーセンタイル・相関行列は十分なデータ量が前提。
各ソースのデータがある期間がバラバラだと、共通で使える期間がごく短くなるリスクがある。
→ まずデータを貯めることが最優先。

## 第3の軸: Curiosity（探索度・意思決定の活発さ）

日付: 2026-03-13

### 背景: 「決められなくなる」という初期兆候

元気がなくなると初期の兆候として**意思決定ができなくなる**。
例: 図書館で毎日4枚CDを選ぶ作業 — ジャンルの開拓、サブスクに無さそうなもの、
といった複雑な判断を楽しんでやっている。これが億劫になったら危険信号。

直接「意思決定力」は計測できないが、**決められなくなった結果として現れる行動パターン**は
データに出る:

1. **選択の冒険度が下がる** — 同じアーティスト・同じジャンルばかり選ぶ（安全圏に逃げる）
2. **選択そのものを避ける** — 4枚借りてたのが3枚、2枚になる。図書館に行かなくなる
3. **全体的にパターンが硬直化する** — 全ソースで「新規」が減る

### 既存モデルの限界: 量だけでは拾えない変化

```
元気な週: 50アーティスト / 200再生 = 探索度 0.25
沈んだ週: 10アーティスト / 200再生 = 探索度 0.05
```

**再生量は同じ200なのに質が違う。** 現行の量ベーススコアリングでは同じスコアになる。
これが「活動はしてるけど中身が変わった」= 意思決定の回避の正体。

### 第3の指標分類: quality

既存の「ベースライン」「イベント」に当てはまらない第3の種類:

| 種類 | ゼロの意味 | 例 |
|------|-----------|-----|
| ベースライン | 異常 | 音楽再生時間、歩数 |
| イベント | 正常（なくてもいい） | ライブ、旅行 |
| **quality** | **活動はあるが質が変わった** | 探索度、ジャンル幅 |

quality 指標は**活動量が正常でも拾える変化**を検知する。

### 1ソースから2つの値を出す

新しいデータソースを追加するのではなく、**既存ソースの別の読み方**として自然に導出できる:

```
lastfm        → Activity: 再生時間     / Curiosity: ユニークアーティスト率
kashidashi    → Activity: 貸出枚数     / Curiosity: 新規アーティスト率
読書メーター   → Activity: 読了冊数     / Curiosity: 新規著者率
Filmarks      → Activity: 鑑賞本数     / Curiosity: 新規監督率
GitHub        → Activity: コミット数   / Curiosity: 新規リポジトリ率
Strava        → Activity: 運動時間     / Curiosity: 新規ルート率
Gmail(EC)     → Activity: 注文件数     / Curiosity: 新規ブランド率
gcal_holiday  → Activity: 予定件数     / Curiosity: 新規スポット率
```

**ほぼ全てのソースから「量」と「探索度」の両方が取れる。**

Last.fm APIでは `user.getWeeklyArtistChart` でアーティスト別再生数が取れるので、
「過去に一度も聴いたことないアーティスト」の割合は算出可能。
Last.fm自体にも新規アーティスト発見数のような指標がある。

### 設計への影響

`source_settings` に `indicator_type: "quality"` を追加し、
同じソースから複数の `source_score` を出せる構造にする。
例: `lastfm` から `lastfm_activity`（量）と `lastfm_curiosity`（探索度）の2行が出る。

### 三軸モデルへの拡張

当初の2軸（健康 × 文化的）では足りないことがわかってきたため、3軸に拡張:

| 軸 | 意味 | 低いと |
|----|------|--------|
| **Activity** | 活動量 | 動けてない |
| **Cultural** | 文化的豊かさ | 文化に触れてない |
| **Curiosity** | 探索・多様性 | 決められない、硬直化 |

### 三次元可視化

3軸 → **Three.js で三次元キューブ表示**。見にくいけど見た目に派手でサイバーな演出に合う。

- **Three.js で半透明のキューブ**、現在位置が光る点（ネオンカラー）
- 時系列の軌跡がネオンの線で残る（過去N週分）
- 各軸の端に CRITICAL / RICH 等のラベル
- キューブの8つの角（状態の組み合わせ）ごとに色が変わる
- 共有ビューでは回転アニメーション、CRITICAL領域に入ると演出が変わる（NERV本部風）
- `@react-three/fiber` で React との統合が容易

本人用では「自分のデータだから文脈がわかる」ので多少読みにくくてもOK。
共有ビューは「雰囲気で状態が伝わる」が目的なのでむしろ3Dが合う。

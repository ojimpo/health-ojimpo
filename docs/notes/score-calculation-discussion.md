# スコア算出方法の議論メモ

日付: 2026-03-12

## 背景

実データを流し込んだ結果、以下の問題が顕在化した:

- **音楽（Last.fm）の存在感が大きすぎる** — ながら聴きで常に積み上がるため、面グラフで他ソースを圧倒する
- **コーディング（トークン数）が爆発する** — トークン数は投入時間と比例しない

## 根本的な気づき: 可処分時間の排他性

データソースの中には「可処分時間を食い合うもの」と「そうでないもの」が混在している。

### 排他的な活動（その間は他のことができない）
- Google Calendar（人と会う）
- Strava ライド（運動中）
- 映画、ライブ

### 並行可能な活動（ながらでできる）
- Last.fm（音楽）— 自転車中も仕事中も聴ける
- Instagram / Twitter（スキマ時間）
- ポッドキャスト

### 食い合いの方向性
- 食い合いはペアごとに異なる（ライドは読書を押し出すが音楽は押し出さない）
- N×Nマトリクスは組み合わせ爆発するため、ソースごとの1次元の係数に落とす方が現実的

## 検討した係数設計

### 案1: 排他性係数を0〜1で設定（並行活動を抑制）
- 音楽: 0.2、ライド: 0.9、カレンダー: 1.0
- **問題**: 既存の自発性係数（spontaneity_coefficient）と掛け合わせると効果が二重になりすぎる
  - 例: lastfm = 0.6(自発性) × 0.2(排他性) = 0.12 ← 潰れすぎ

### 案2: 排他性ボーナス（1.0以上の値で排他的活動を持ち上げる）
- 音楽: 1.0、ライド: 1.2、カレンダー: 1.3
- **利点**: 自発性係数と掛けても下方に二重で潰れない
- **問題**: 音楽の絶対値が大きい問題は解決しない（他を持ち上げても音楽の面積は変わらない）

### 案3: 1.0を基準に上下に振る（0.x 〜 1.x）
- 並行活動は1.0未満（軽く抑える）、排他活動は1.0以上（軽くボーナス）
- **問題**: 結局自発性係数との複合効果の懸念が残る

## 未解決事項: 自発性係数の妥当性

- spontaneity_coefficient はまだ有用かどうか判断がついていない
- 妥当性が不明な係数の上にさらに係数を重ねると、何が効いているか切り分けられなくなるリスク

## 今後の方針: 対話的な係数チューニング

### アプローチ: Human-in-the-loop での探索

事前に正解ラベルをまとめるのではなく、**AIと対話しながら係数を調整する**:

1. AI が係数の組み合わせを変えてスコア・グラフを算出して提示
2. 本人がグラフを見て「この時期は低調だったはずなのに高い」「ここは合ってる」とフィードバック
3. AI がフィードバックをもとに係数を調整して再提示
4. 繰り返して収束させる

**この方法の利点:**
- グラフを見ながら記憶を掘り起こせる（事前にラベルを網羅する必要がない）
- 「なんとなく違う」という曖昧な感覚も拾える
- 自発性係数と排他性係数のどちらが効いているか、オン/オフしながら切り分けられる

### 実装に必要なもの
- 係数を変えた場合のスコアを再計算する仕組み（過去データ全期間）
- 結果をグラフとして提示する手段
- 係数セットの保存・比較機能（「さっきの方がよかった」に対応）

### 別案: 正解ラベル方式による自動探索（Claude提案）

Human-in-the-loop と併用または単独で使える手法:

1. 過去の特定期間に「正解ラベル」を与える（低調/普通/充実 等）
2. ラベルとスコアの相関が最大になる係数の組み合わせをAIが自動探索
3. 数年分のデータ × 数十点のラベルがあれば十分な精度で探索可能

```
正解ラベルの例:
2024-03:  低調（仕事が忙しくて何もできなかった）
2024-08:  充実（夏休みでライブ3本、本も読んだ）
2024-12:  普通
```

**利点:** 対話的チューニングで得たフィードバックをラベルとして蓄積し、後から自動探索に回すこともできる。両方のアプローチは補完関係にある。

## 技術的負債: minutes カラムの問題

当初「すべて分換算」を目指して `activity_records.minutes` を設けたが、実際には分に変換できないソースが多い:

| ソース | minutes の中身 | 実態 |
|--------|---------------|------|
| Last.fm, Strava, SNS | 実際の時間(分) | 自然 |
| Oura readiness/sleep | スコア(0-100) | スコアを分として格納 |
| kashidashi_cd | 枚数 | 3枚 = 3分？ |
| intervals.icu, Gmail, OpenAI/Claude, GitHub, gcal | 0 | 分に変換不能 |

**現状のスコアリングの fallback:**
```python
SUM(CASE WHEN minutes > 0 THEN minutes ELSE raw_value END)
```

基準値で正規化されるため計算結果自体は壊れていないが、コードの意味が曖昧。
**修正案:** スコアリングで `raw_value` を一貫して使い、`minutes` は参考情報に留める。

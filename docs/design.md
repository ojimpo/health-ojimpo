# 文化的生活ダッシュボード 設計ドキュメント v0.5

2026年3月

---

## 1. コンセプト

日常の文化的活動（読書、音楽、映画、運動、SNS閲覧、コーディング、買い物など）のデータを外部サービスから自動取得し、共通のスコアとして数値化する。スコアの推移をモニタリングすることで、文化的生活度の変化、ひいてはメンタルヘルスの状態変化を早期に察知する。

本人が確認するだけでなく、親しい友人にも共有可能とすることで、セーフティーネットとして機能させることを目的とする。

単なるライフログアプリではなく、「自分の元気さ」「生活の豊かさ」「余裕」を可視化するためのダッシュボードである。

### 設計原則

- **入力ゼロ**: 本ダッシュボードへの直接入力は行わない。すべてのデータは既存の外部サービスから自動取得する。
- **指標の柔軟性**: 指標はあとから自由に追加・削除でき、それによって過去のスコアが崩れない設計とする。
- **可処分時間の制約を考慮**: 読書をしまくって映画を見ない週も健康。活動同士が限られた可処分時間を奪い合う関係にあることを前提とし、特定の活動に集中していること自体はネガティブではない。

## 2. スコアリングモデル

### 2.1 指標の2分類

すべての指標を以下の2種類に分類する。この分類は管理画面から変更可能。

| 分類 | 性質 | スコアへの影響 | 例 |
|------|------|---------------|-----|
| **ベースライン指標** | 日常的にほぼ必ず発生する。ゼロになること自体が異常の兆候 | 減少・消失がスコアを**下げる** | 音楽再生、Instagram閲覧、Twitter閲覧、通勤記録、睡眠、体重測定、買い物 |
| **イベント指標** | やるときはやるが、ゼロの期間があっても正常 | 発生時にスコアを**加算**（ゼロでも減点なし） | 読了、映画鑑賞、休日の予定、休日ライド、図書館貸出、音楽ライブ、コーディング活動 |

### 2.2 指標の2カテゴリ（表示用）

グラフ表示においては、指標を以下の2カテゴリに分ける。

| カテゴリ | 性質 | グラフ表示 | 例 |
|---------|------|----------|-----|
| **活動量系** | 「どれだけやったか」を示す累積的な指標 | 積み上げ面グラフ | 音楽再生時間、運動距離、読書冊数、映画鑑賞、SNS使用時間、コーディング活動 |
| **状態系** | 「今どういう状態か」を示す時点の指標 | 折れ線グラフ（重ね表示） | 体重、睡眠スコア、Readiness Score、Stress Score、CTL/FTP |

### 2.3 共通データスキーマ

各データソースから取得した生データは、以下の共通スキーマに変換して保存する。

```
{
  date:        日付
  source:      データソース名（例: "lastfm", "strava", "oura"）
  category:    活動カテゴリ（例: "music", "exercise", "sleep", "shopping"）
  minutes:     活動時間（分）※ 推定値の場合あり
  raw_value:   元データの生値（例: 再生回数、距離km、睡眠スコア、注文件数）
  raw_unit:    生値の単位（例: "tracks", "km", "score", "orders"）
  metadata:    付加情報（例: Stravaのcommuteフラグ、曜日種別など）
}
```

### 2.4 正規化（基準値方式）

各指標に「基準値」を設定し、基準値に対するパーセンテージとしてスコア化する。100点を「いい感じ」の基準とし、上限は設けない。

**基準値の設定方法：**

```
読書（読書メーター）
  期間: 月間
  基準値: 7冊 = 100点
  
自転車（Strava）
  期間: 月間
  基準値: 200km = 100点
```

**基準値には適用期間とメモを持たせる（intervals.icuのFTP管理と同様）：**

```
自転車（Strava）
  2026-03-01〜  月間200km = 100点  メモ: 自転車通勤を開始
  2026-09-01〜  月間300km = 100点  メモ: 通勤距離が伸びた
```

過去のスコアは当時の基準値で計算されるため、基準変更が過去データを歪めない。

**スコアの意味：**
- 100 = 基準（自分にとっての「いい感じ」）
- 150 = 基準の1.5倍（絶好調）
- 50 = 基準の半分（注意）
- 0付近 = 活動停止

**自動モード（将来）：** 過去最高の月間実績を基準値として自動設定する。過去最高を更新したら基準が上がるだけで、過去のデータは当時の基準でそのまま残る。

**初期基準値の設定例（肌感）：**

| 指標 | 期間 | 基準値 | 100点の意味 |
|------|------|--------|-----------|
| 読書 | 月間 | 7冊 | 月7冊読了 |
| 自転車 | 月間 | 200km | 月200km走行 |
| 他の指標 | — | 運用しながら設定 | — |

**指標間の比較：** 各指標の100点基準を設定することで、指標間の等価関係が暗黙的に成立する。新しい指標を追加する際は「この指標の100点は何か」を1つ決めるだけで、既存指標への影響はない。

### 2.5 基本算出式

**ベースラインスコア（健康指標の判定に使用）：**

```
各ベースライン指標のスコア = (raw_value / 基準値) × 100 × 自発性係数
↓
ベースラインスコア = 全ベースライン指標の加重平均
  ※ゼロの指標は0点として平均に含める（除外しない）
  ※Instagram消失 → 0点が平均に含まれ、スコアが明確に低下
```

**活動量スコア（文化的指標の判定に使用）：**

```
各活動量系指標のスコア = (raw_value / 基準値) × 100 × 自発性係数
↓
活動量スコア = 発生した活動量系指標のスコア合計
  ※イベント指標は未発生なら0で、合計に加算されないだけ（ペナルティなし）
  ※積み上げ面グラフの面積と一致
```

上限を設けないことで、絶好調の時期が積み上げ面グラフで面積が大きくなり、視覚的に伝わる。

**健康指標の判定閾値（初期値、管理画面で調整可能）：**

| ステータス | ベースラインスコア |
|-----------|----------------|
| NORMAL | 70以上 |
| CAUTION | 40〜70 |
| CRITICAL | 40未満 |

**文化的指標の判定閾値（初期値、管理画面で調整可能）：**

| ステータス | 活動量スコア（基準比） |
|-----------|-------------------|
| RICH | 基準の70%以上 |
| MODERATE | 基準の40〜70% |
| LOW | 基準の40%未満 |

### 2.6 自発性係数

| 自発性レベル | 係数（初期値） | 例 |
|-------------|--------------|-----|
| 義務的 | 0.3 | 通勤サイクリング（Strava commuteフラグ） |
| 半自発的 | 0.6 | BGM的な音楽再生 |
| 自発的 | 1.0 | 休日のライド、読書、映画鑑賞 |
| 高自発的 | 1.2 | 休日の外出予定、音楽ライブ |

### 2.7 集計期間（移動平均の窓 = 基準値の期間）

各指標の集計期間は移動平均の窓と基準値の期間を兼ねる。管理画面の各ソース設定から調整可能。

```
音楽（Last.fm）
  集計期間: 7日間
  基準値: 期間あたり700分再生 = 100点（≒1日100分）

ライブ（Googleカレンダー）
  集計期間: 90日間
  基準値: 期間あたり3回 = 100点（≒月1回ペース）

読書（読書メーター）
  集計期間: 90日間
  基準値: 期間あたり21冊 = 100点（≒月7冊ペース）

自転車（Strava）
  集計期間: 30日間
  基準値: 期間あたり200km = 100点
```

集計期間を長く取ることで、ライブのように「月1が目標だが実際は偏りがある」指標（1月に2回行って2月は0回でもOK）を自然に扱える。読書のように1冊に時間がかかる指標も、長い集計期間で平滑化される。

## 3. データソース

### 3.1 データ取得方式

| 取得方式 | 説明 | 対象 |
|---------|------|------|
| **公式API** | サービスが提供するREST APIから直接取得 | Last.fm, Strava, Googleカレンダー, Gmail, Oura, GitHub, Anthropic Admin API, OpenAI Usage API, intervals.icu |
| **Screen Time Network** | Clearspaceアプリ経由でiOSスクリーンタイムデータをAPI取得 | Instagram, Twitter |
| **自作API + OpenClawスクレイピング** | 専用APIアプリを構築し、OpenClawでスクレイピングしたデータを投入（kashidashiと同パターン） | 読書メーター, Filmarks |
| **自作API（稼働済み）** | 同一サーバー上で稼働中の自作アプリからAPI取得 | kashidashi |

取得はバッチ処理（1日1回程度）で行う。取得した生データはサービスごとのrawテーブルに保存し、共通スキーマへの変換後にスコアリングに使用する。

### 3.3 フェーズ戦略

| フェーズ | 目的 | データソース | 選定理由 |
|---------|------|------------|---------|
| **MVP** | 取得→保存→スコアリング→表示の一連の流れを通す | Last.fm | 公式APIあり。日次で細やかにデータが変動し、1ソースでパイプライン全体を検証できる |
| **Phase 2** | 日次データの指標を拡充し、スコアの感度を上げる | Strava, Oura, intervals.icu, Instagram, Twitter, Googleカレンダー, Gmail, kashidashi | 日次でデータが出る、またはAPI構築済み。Stravaはcommuteタグ自動付与の仕組みが整い次第追加。kashidashiは自作APIが稼働済み |
| **Phase 3** | スクレイピング系・低頻度・要調査の指標を追加 | 読書メーター, Filmarks, GitHub, Anthropic, OpenAI, スマートバスマット, Spotify（ポッドキャスト） | スクレイピング実装が必要、低頻度、API調査が必要 |

**MVP検証方針：** Last.fmの過去数年分の再生履歴を一括取得し、直近の病み期がグラフ上で「しぼむ」ことを確認する。これによりスコアリングモデルの妥当性を初期段階で実証する。

**設定画面のUI方針：** MVP段階から全データソース（17個）を設定画面に表示する。未実装のソースはグレーアウトで「Coming Soon」と表示し、全体像とロードマップが常に見えるようにする。

### 3.4 将来の拡張候補

美術館・博物館の訪問記録、ゲームプレイ時間（Steam等）、CAD活動時間（Screen Time Network経由でアプリ別使用時間を合算。ただしWindows側は課題あり）、3Dプリンタ稼働時間（Flashforge ADV3は非公式TCP APIでステータス取得可能。LAN内からポーリングで印刷累積時間を記録）など。データソースは今後も自由に追加可能な設計とする。

## 4. 表示仕様

### 4.1 メインビジュアル（2段構成）

**上段：積み上げ面グラフ（活動量系）**
- 音楽再生、運動、読書、映画、SNS、コーディング等の活動量を積み上げ表示
- 100点の基準ラインを薄く表示し、基準との比較が一目でわかるようにする
- 各面の色分け：データソースまたは活動カテゴリごと
- 折れ点に小さなドットを表示
- 線はスムージングなし（角あり）

**下段：折れ線グラフ（状態系）**
- 体重、睡眠スコア、Readiness Score、Stress Score、CTL/FTP等を重ねて表示
- 各指標を色分けした折れ線で表示

**共通仕様：**
- 横軸：時間（直近3ヶ月、週単位）
- デザイン：ダークカラー基調、ネオンカラーが黒背景に映えるサイバーな雰囲気
- URL：health.ojimpo.com
- タイトル表記：HEALTH.OJIMPO.COM

上段で「何をどれだけやったか」、下段で「結果としてどういう状態か」。2つを並べることで「活動量が減った → 睡眠スコアも落ちてきた」のような因果関係も読み取れる。

### 4.2 表示期間の切り替え

| 期間 | 粒度 | 用途 |
|------|------|------|
| 直近1ヶ月 | 日単位 | 短期の変動を詳細に確認 |
| 直近3ヶ月 | 週単位 | **デフォルト**。中期の傾向把握 |
| 直近1年 | 月単位 | 長期の傾向・季節変動の確認 |

### 4.3 傾向コメントの自動生成

グラフに加えて、現在の状態を自然言語で要約するコメントを自動生成する。本人用・共有ビューの両方に表示する。

生成例：
- 「今週は読書が多く、映画は少なめ。文化活動の総量は安定しています」
- 「運動量は維持されていますが、音楽・SNSの活動が2週連続で低下しています」
- 「文化活動総量が3週連続で低下しています」
- 「今月はまだ音楽ライブに行っていません」

### 4.4 最近の活動フィード

直近の活動を時系列で表示するセクション。本人用ビューでは詳細あり、共有ビューでは抽象化して表示。

本人用表示例：
- 1日前　1冊の本を読了 — 村上春樹『街とその不確かな壁』
- 1日前　音楽を2時間34分再生 — 32トラック
- 2日前　休日ランニング 5.2km — Strava記録

共有ビュー表示例：
- 1日前　1冊の本を読了
- 1日前　音楽を2時間34分再生
- 2日前　休日ランニング 5.2km

### 4.5 比較表示

- **週次比較**: 先週との差分を表示
- **月次比較**: 先月との差分を表示
- **「元気だった時期」との比較**: 基準値の自動モードと連動

### 4.6 ビュー

| ビュー | 対象 | 説明 |
|-------|------|------|
| 本人用ダッシュボード | 本人のみ | 全データの詳細表示、比較表示、管理画面へのアクセス、活動フィードに詳細あり |
| 共有ビュー | 友人 | 健康状態の表示、積み上げ面グラフ、傾向コメント、活動フィード（抽象化）を表示 |

### 4.7 ステータス表示（両ビュー共通）

両ビューの目立つ位置に、健康指標と文化的指標を横に並べて表示する。

```
[NORMAL ●]  [RICH ●]
 健康状態     文化活動
```

**健康指標（ベースライン由来）：**

| ステータス | 判定基準 | 表示 |
|-----------|---------|------|
| NORMAL | ベースライン指標の平均が基準以上 | 「健康だと思われるのでいつも通り接して大丈夫です」 |
| CAUTION | ベースライン指標の平均がやや低下 | 「少し気にかけてあげてください」 |
| CRITICAL | ベースライン指標の平均が大幅低下、または消失あり | 「連絡を取ってみてください」 |

**文化的指標（活動量由来）：**

| ステータス | 判定基準 | 表示 |
|-----------|---------|------|
| RICH | 活動量系指標の合計が基準以上 | 文化活動が豊か |
| MODERATE | 活動量系指標の合計がやや低下 | 文化活動がやや控えめ |
| LOW | 活動量系指標の合計が大幅低下 | 文化活動がほぼ停止 |

**演出マトリクス：**

| 健康指標 | 文化的指標 | 演出 |
|---------|----------|------|
| NORMAL | RICH | 緑 + ネオンカラー（通常） |
| NORMAL | MODERATE | 緑 + やや控えめな色味 |
| NORMAL | LOW | 緑 + グラフがモノクロ化 |
| CAUTION | RICH | オレンジ + ネオンカラー |
| CAUTION | MODERATE | オレンジ + やや控えめ |
| CAUTION | LOW | オレンジ + モノクロ化 |
| CRITICAL | RICH | エヴァ暴走 + ネオンカラー |
| CRITICAL | MODERATE | エヴァ暴走 + やや控えめ |
| CRITICAL | LOW | **エヴァ暴走 + モノクロ化（最悪状態）** |

各指標の閾値は管理画面から調整可能とする。

## 6. 管理画面（本人専用）

### 6.1 ソース一覧

全データソースを一覧表示し、以下をソースごとにトグル・設定できる。

```
音楽（Last.fm）          [本人 ✓] [共有 ✓] [設定...]
運動（Strava）           [本人 ✓] [共有 ✓] [設定...]
睡眠（Oura）             [本人 ✓] [共有 ○] [設定...]
体重（スマートバスマット）  [本人 ✓] [共有 ✗] [設定...]
...
```

### 6.2 各指標の設定

各指標の[設定...]から以下を調整可能。

- 「ベースライン / イベント」分類の変更
- 「活動量 / 状態」カテゴリの変更
- 基礎ポイントの設定
- 自発性係数の設定
- 移動平均の窓サイズの調整

### 6.3 基準値の管理

各指標の「100点の基準値」を期間付きで管理する。intervals.icuのFTP管理と同様のUI。

```
自転車（Strava）
  2026-03-01〜  月間200km = 100点  メモ: 自転車通勤を開始
  [+ 新しい基準値を追加]
  
  □ 自動モード: 過去最高月間を基準値に設定
```

### 6.4 全体設定

- 健康状態ステータスの閾値設定
- 共有ビューの公開URL管理

## 8. 将来構想

### 8.1 スコア重み付けの自動最適化

十分なデータが蓄積された段階で、本人が「調子がいい」と感じた時期のデータパターンを学習し、それに近い状態を高スコアとするように重みを自動調整する仕組みを検討する。

- 前提条件: 数ヶ月以上の継続的なデータ蓄積
- 入力: 定期的な主観的コンディション評価（簡易な5段階評価など）
- 出力: 最適化された基礎ポイント・自発性係数の提案
- 応用: 「元気だった時期」の特徴量を抽出し、比較表示の基準とする

### 8.2 アラート機能

スコアが一定期間にわたり閾値を下回った場合に、本人や共有先の友人に通知を送る機能。

### 8.3 モバイル展開

将来的にモバイルアプリまたはPWAとして展開し、外出先からも確認できるようにする。

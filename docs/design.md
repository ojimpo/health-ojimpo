# 文化的生活ダッシュボード 設計ドキュメント v0.6

2026年4月

---

## 1. コンセプト

日常の文化的活動（読書、音楽、映画、運動、SNS閲覧、コーディング、買い物など）のデータを外部サービスから自動取得し、共通のスコアとして数値化する。スコアの推移をモニタリングすることで、文化的生活度の変化、ひいてはメンタルヘルスの状態変化を早期に察知する。

本人が確認するだけでなく、親しい友人にも共有可能とすることで、セーフティーネットとして機能させることを目的とする。

単なるライフログアプリではなく、「自分の元気さ」「生活の豊かさ」「余裕」を可視化するためのダッシュボードである。

### 察知から行動変容へ

このダッシュボードの当初の目的はメンタルヘルスの変化を「察知」することだったが、運用を続けるうちに副次的な効果が見えてきた。ダッシュボードに映る自分の姿を良くしたいという動機が、実際の行動変容を促すのである。

- 買い物指標を入れたくて家計簿を始める
- 読書メーターのスコアが気になって本を手に取る
- Stravaの活動量を維持したくて運動を続ける

「計測するから行動する」というフィードバックループが自然に生まれ、ダッシュボードが生活のコーチのような役割を果たし始める。データソースを追加することは単なる機能拡張ではなく、生活の中に新しい習慣を組み込むきっかけになる。これは設計時には意図していなかったが、このアプリの本質的な価値かもしれない。

### 設計原則

- **入力ゼロ**: 本ダッシュボードへの直接入力は行わない。すべてのデータは既存の外部サービスから自動取得する。
- **指標の柔軟性**: 指標はあとから自由に追加・削除でき、それによって過去のスコアが崩れない設計とする。
- **可処分時間の制約を考慮**: 読書をしまくって映画を見ない週も健康。活動同士が限られた可処分時間を奪い合う関係にあることを前提とし、特定の活動に集中していること自体はネガティブではない。
- **1カテゴリ1ソース原則**: 性質の異なるソースは別カテゴリに分ける。複数ソースを1カテゴリにまとめるとスコアが膨らみ、グラフのバランスが崩れる。

## 2. スコアリングモデル

### 2.1 指標の分類

すべての指標を以下の種類に分類する。この分類は管理画面から変更可能。

| 分類 | 性質 | スコアへの影響 | 例 |
|------|------|---------------|-----|
| **ベースライン** | 日常的にほぼ必ず発生する。ゼロになること自体が異常の兆候 | 減少・消失がスコアを**下げる** | 音楽再生、SNS閲覧、睡眠、買い物 |
| **イベント** | やるときはやるが、ゼロの期間があっても正常 | 発生時にスコアを**加算**（ゼロでも減点なし） | 読了、映画鑑賞、休日の予定、音楽ライブ、CD貸出、コーディング活動 |
| **health_only** | 健康スコアの判定にのみ参加、グラフ・文化スコアには不参加 | ベースライン平均に含まれる | 外出推定（NextDNS cellular比率） |

### 2.2 表示タイプ（display_type）

指標の表示方法を制御する。スコア計算への参加と独立した概念。

| display_type | グラフ表示 | カテゴリカード | 文化スコア参加 | 用途 |
|-------------|----------|-------------|-------------|------|
| **activity** | 積み上げ面グラフに表示 | 表示 | 参加 | 通常の活動量系指標 |
| **card_only** | グラフ非表示 | 表示 | 参加 | スケールが他と乖離する指標（コード系） |
| **state** | CONDITIONタブの折れ線 | 状態カードとして表示 | 不参加 | 睡眠、CTL等の状態系指標 |

**card_only導入の背景**: Claude Codeのトークン量（日次最大143k tokens）が他カテゴリの10-20倍に達し、積み上げグラフのスケールを破壊していた。コーディングは趣味の文化的活動であり文化スコアには参加させたいが、グラフでは他カテゴリが見えなくなるため、グラフからは除外しカード表示のみとした。

### 2.3 共通データスキーマ

各データソースから取得した生データは、以下の共通スキーマに変換して保存する。

```
{
  date:        日付
  source:      データソース名（例: "lastfm", "strava", "oura"）
  category:    活動カテゴリ（例: "music", "exercise", "sleep", "shopping"）
  minutes:     活動時間（分）※ 推定値の場合あり
  raw_value:   元データの生値（例: 再生回数、距離km、睡眠スコア、注文件数）
  raw_unit:    生値の単位（例: "tracks", "km", "score", "orders"）
  metadata:    付加情報（例: 曜日種別など）
}
```

### 2.4 正規化（基準値方式）

各指標に「基準値」を設定し、基準値に対するパーセンテージとしてスコア化する。100点を「いい感じ」の基準とし、上限は設けない。

**基準値には適用期間とメモを持たせる（intervals.icuのFTP管理と同様）：**

```
自転車（Strava）
  2026-03-01〜  月間200km = 100点  メモ: 自転車通勤を開始
  2026-09-01〜  月間300km = 100点  メモ: 通勤距離が伸びた
```

過去のスコアは当時の基準値で計算されるため、基準変更が過去データを歪めない。

**スコアの意味：**
- 100 = 基準（自分にとっての「いい感じ」）
- 150 = 基準の1.5倍（絶好調）
- 50 = 基準の半分（注意）
- 0付近 = 活動停止

### 2.5 指数減衰（decay_half_life）

イベント型ソースのスパイクを平滑化するため、窓切断方式に代わり指数減衰を採用。

```
score = Σ(raw_value_i × e^(-ln2 × days_ago_i / half_life)) / norm_base × 100
norm_base = (base_value / period) × (half_life / ln2)
```

- half_life: 7日系ソース=7日、90日系=30日
- 効果: 昨日の活動と3ヶ月前の活動に適切な重み差をつける
- 窓方式の問題（N日前に急にスコアがリセットされる）を解消

### 2.6 基本算出式

**ベースラインスコア（健康指標の判定に使用）：**

```
各ベースライン指標のスコア = (raw_value / 基準値) × 100 × 自発性係数
↓
ベースラインスコア = 全ベースライン指標の加重平均
  ※ゼロの指標は0点として平均に含める（除外しない）
```

**活動量スコア（文化的指標の判定に使用）：**

```
各活動量系指標のスコア = (raw_value / 基準値) × 100 × 自発性係数
↓
活動量スコア = display_type IN ('activity', 'card_only') のスコア合計
  ※イベント指標は未発生なら0で、合計に加算されないだけ（ペナルティなし）
```

上限を設けないことで、絶好調の時期が積み上げ面グラフで面積が大きくなり、視覚的に伝わる。

**健康指標の判定閾値（初期値、管理画面で調整可能）：**

| ステータス | ベースラインスコア |
|-----------|----------------|
| NORMAL | 70以上 |
| CAUTION | 40〜70 |
| CRITICAL | 40未満 |

**文化的指標の判定閾値（初期値、管理画面で調整可能）：**

| ステータス | 活動量スコア（基準比） |
|-----------|-------------------|
| RICH | 基準の70%以上 |
| MODERATE | 基準の40〜70% |
| LOW | 基準の40%未満 |

### 2.7 自発性係数

| 自発性レベル | 係数（初期値） | 例 |
|-------------|--------------|-----|
| 半自発的 | 0.6 | BGM的な音楽再生 |
| 自発的 | 1.0 | 運動、読書、映画鑑賞 |
| 高自発的 | 1.2 | 休日の外出予定、音楽ライブ |

### 2.8 集計期間（基準値の期間）

各指標の集計期間は基準値の期間と一致する。管理画面の各ソース設定から調整可能。

```
音楽（Last.fm）    集計期間: 7日間   基準値: 1200min = 100点
運動（Strava）     集計期間: 7日間   基準値: 180min = 100点  decay_half_life: 7
CD貸出（kashidashi）集計期間: 7日間   基準値: 16枚 = 100点   decay_half_life: 7
ライブ（Gcal）     集計期間: 90日間  基準値: 3回 = 100点     decay_half_life: 30
読書（読書メーター）集計期間: 90日間  基準値: 21冊 = 100点    decay_half_life: 30
```

集計期間を長く取ることで、ライブのように「月1が目標だが実際は偏りがある」指標を自然に扱える。

## 3. データソース

### 3.1 データ取得方式

| 取得方式 | 説明 | 対象 |
|---------|------|------|
| **公式API** | サービスが提供するREST APIから直接取得 | Last.fm, Strava, Googleカレンダー, Oura, GitHub, intervals.icu |
| **NextDNS API** | DNSクエリ数からサービス利用量を推定 | SNS, 買い物, 活力(Vitality), 外出推定 |
| **sync-gateway経由** | OpenClawがスクレイピングしたデータをsync-gateway APIから取得 | 読書メーター, Filmarks |
| **自作API** | 同一サーバー上で稼働中の自作アプリからAPI取得 | kashidashi |
| **ローカル集計** | NAS上のローカルファイルを直接集計 | Claude Code（JSONL）, Stash（GraphQL） |

### 3.3 設計判断の記録

**Strava一本化（2026-04-06）**: 当初はstrava（全運動）、strava_voluntary（自発的運動）、strava_ride（ライド距離）、strava_commute（通勤）の4ソースに分離していたが、運動の種類や動機は健康指標として無関係であるため、stravaのみに統一。decay_half_life=7で日次スパイクを平滑化。

**coding card_only化（2026-04-06）**: Claude Codeのトークン量が他カテゴリの10-20倍に達し積み上げグラフのスケールを破壊していた。趣味コーディングは文化的活動なので文化スコアには参加させつつ、グラフからは除外。

**kashidashi独立カテゴリ化（2026-04-06）**: 音楽（Last.fm、聴く行為）とCD貸出（kashidashi、図書館で借りる行為）は性質が異なるため、musicからcdカテゴリに分離。base_value=16（週4回×4枚）、decay_half_life=7。

### 3.4 将来の拡張候補

美術館・博物館の訪問記録、ゲームプレイ時間（Steam等）、体重（iOSヘルスケア）、Spotify Podcast等。データソースは今後も自由に追加可能な設計とする。

## 4. 表示仕様

### 4.1 グラフ（3モードタブ）

| モード | 積み上げ面グラフ | 折れ線 | 用途 |
|-------|---------------|-------|------|
| **ACTIVITY** | カラフル（各カテゴリ色） | なし | デフォルト。活動量の推移 |
| **SCORE** | モノクロ | 健康/文化スコア折れ線（ステータス色で変化） | スコア推移の確認 |
| **CONDITION** | モノクロ | sleep/readiness/stress/outing/CTL折れ線 | 身体状態の確認 |

**積み上げ面グラフ（全モード共通の背景）:**
- display_type=activityのカテゴリのみ表示（card_only/stateは非表示）
- 100点の基準ラインを薄く表示
- 折れ点に小さなドットを表示
- 線はスムージングなし（角あり）
- Y軸: tickを手動制御（中央値ベース）、domain=[0, dataMax]でスパイクも表示

### 4.2 表示期間の切り替え

| 期間 | 粒度 | データポイント数 |
|------|------|---------------|
| 直近1ヶ月 | 日単位 | 30 |
| 直近3ヶ月 | 日単位 | 90 |
| 直近1年 | 週単位 | 52 |

### 4.3 カテゴリカード

各カテゴリのスコアをカード形式で表示。display_type=activityとcard_onlyの両方を表示。
先週との差分（+/-）を色付きで表示。

### 4.4 傾向コメントの自動生成

グラフに加えて、現在の状態を自然言語で要約するコメントを自動生成する。本人用・共有ビューの両方に表示する。

生成例：
- 「今週は読書が多く、映画は少なめ。文化活動の総量は安定しています」
- 「運動量は維持されていますが、音楽・SNSの活動が2週連続で低下しています」

### 4.5 最近の活動フィード

直近の活動を時系列で表示するセクション。本人用ビューでは詳細あり、共有ビューでは抽象化して表示。

### 4.6 ビュー

| ビュー | URL | 対象 | 説明 |
|-------|-----|------|------|
| 共有ビュー | `/` | 誰でも | 健康状態の表示、積み上げ面グラフ、傾向コメント、活動フィード（抽象化） |
| 本人用ダッシュボード | `/admin` | 本人のみ（CF Access） | 全データの詳細表示、比較表示、活動フィードに詳細あり |
| 設定画面 | `/settings` | 本人のみ（CF Access） | ソース設定、閾値設定、基準値管理 |

### 4.7 ステータス表示（両ビュー共通）

両ビューの目立つ位置に、健康指標と文化的指標を横に並べて表示する。

**演出マトリクス：**

| 健康指標 | 文化的指標 | 演出 |
|---------|----------|------|
| NORMAL | RICH | 緑 + ネオンカラー（通常） |
| NORMAL | MODERATE | 緑 + やや控えめな色味 |
| NORMAL | LOW | 緑 + グラフがモノクロ化 |
| CAUTION | RICH | オレンジ + ネオンカラー |
| CAUTION | MODERATE | オレンジ + やや控えめ |
| CAUTION | LOW | オレンジ + モノクロ化 |
| CRITICAL | RICH | エヴァ暴走 + ネオンカラー |
| CRITICAL | MODERATE | エヴァ暴走 + やや控えめ |
| CRITICAL | LOW | **エヴァ暴走 + モノクロ化（最悪状態）** |

| データソース | 本人用表示 | 共有ビュー表示 |

| Last.fm | 再生履歴・時間・スコア | スコアのみ |
| Strava | 運動内容・距離・スコア | スコアのみ |

| NextDNS SNS | クエリ数・スコア | スコアのみ |
| GitHub / Claude Code | 活動量・スコア | スコアのみ |

各データソースの共有可否は管理画面のソース一覧からトグルで切り替え可能。

## 8. 将来構想

### 8.1 スコア重み付けの自動最適化

十分なデータが蓄積された段階で、本人が「調子がいい」と感じた時期のデータパターンを学習し、それに近い状態を高スコアとするように重みを自動調整する仕組みを検討する。

### 8.2 通知機能（実装済み・有効化待ち）

コンディション低下時に購読者へLINE / メールで自動通知する機能。6時間ごとのingestバッチ完了後にステータス遷移を検出し通知を送信する。

**通知トリガー：**

| 遷移 | 通知 |
|------|------|
| Health: NORMAL → CAUTION | する |
| Health: NORMAL → CRITICAL | する |
| Health: CAUTION → CRITICAL | する |
| Cultural: → LOW（RICH/MODERATEから） | する |
| Cultural: RICH → MODERATE | しない（静観） |
| 回復方向（CAUTION → NORMAL 等） | しない |
| 同じ状態が続く | しない |

### 8.3 Curiosity軸（第3指標構想）

活動量の「質」を検知する第3軸として、既存ソースの別の読み方から導出する構想。例: GitHubの新規リポジトリ率で「探索度」を検知。新ソース不要で実装可能。
